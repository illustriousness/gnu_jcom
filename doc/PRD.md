# 关于类 JCom 串口工具在 Linux / Wayland 上实现的讨论整理

## 1. 需求背景

目标是参考 **JCom 串口工具** 的能力，在 **Linux** 上实现一套类似功能的桌面工具。  
当前优先级明确为：

1. **Linux 优先**
2. **优先支持 Wayland**
3. 功能实现稳定后，再考虑 **X11**
4. 后续再评估其他平台适配

---

## 2. 对 JCom 这类工具的理解

JCom 这类工具的价值，并不只是“串口收发”，而是把几种能力整合成了一个完整的调试工作流：

- 串口基础通信
- 命令分组和批量发送
- 协议字段级组包与解析
- 自动校验（CRC / Checksum）
- 控件化交互（按钮、开关、滑条等）
- 实时曲线和数据面板
- 工程配置保存与复用

也就是说，它本质上更接近：

> **协议化的设备调试平台**
>
> 而不只是一个传统串口助手。

所以在实现时，最值得复刻的是它的**产品思路**，而不是某个界面长相。

---

## 3. 功能拆分建议

为了避免一开始把系统做得过重，可以把需求拆成三层。

### 3.1 终端层

这一层是所有功能的底座，至少应包含：

- 串口枚举
- 打开 / 关闭串口
- 波特率、数据位、校验位、停止位、流控配置
- ASCII / HEX 显示与发送
- 定时发送
- 日志保存
- DTR / RTS 控制
- 热插拔检测
- 自动重连

这部分是基础，没有它后面都站不住。

---

### 3.2 协议层

这是类 JCom 工具和普通串口助手拉开差距的关键。

建议支持：

- 自定义帧结构
- 字段类型定义（u8/u16/i16/f32/bits/string/bytes）
- 大小端设置
- 长度字段自动联动
- 校验字段自动更新
- 公式计算 / 派生字段
- 接收侧协议识别
- 粘包 / 分包处理

如果协议层设计得好，工具会从“终端”升级成“协议工作台”。

---

### 3.3 交互与可视化层

这一层决定最终体验：

- 字段编辑器
- 命令模板面板
- 初始化命令列表
- 控件化发送（按钮、开关、滑条、枚举）
- 实时曲线
- 数据表格
- 阈值告警
- 导出 CSV / JSON
- 工程文件保存与恢复

---

## 4. 初步技术选型判断（跨平台视角）

如果单纯从“跨平台桌面应用”角度看，比较合理的候选方案主要有三类。

### 4.1 Rust + Tauri

优点：

- 跨平台能力较好
- Rust 适合做串口 I/O、协议解析等底层逻辑
- 前端用 Web 技术实现 UI，开发效率高
- 包体和资源占用通常优于 Electron

适合：

- 个人或小团队
- 追求迭代速度
- 同时考虑后续 Windows/macOS

---

### 4.2 Qt 6 + QSerialPort

优点：

- 串口和桌面 GUI 都很成熟
- 工业工具场景经验丰富
- 原生桌面体验好
- 很适合长期做复杂工程工具

适合：

- 团队熟悉 Qt
- 目标是桌面原生工具
- 未来还会扩展更多工业协议或设备支持

---

### 4.3 Python（PySide / pySerial）

优点：

- 原型验证快
- 内部工具开发效率高

问题：

- 长期工程化和分发能力较弱
- 高频实时曲线、长时间运行、复杂项目维护时不如 Rust / Qt 稳定

适合：

- 原型验证
- 内部调试工具
- 快速试错

---

## 5. 当目标调整为“Linux 优先 + Wayland 优先”后，选型判断变化

当需求明确成：

- Linux 优先
- Wayland 第一优先级
- 功能稳定后再补 X11

那么技术选型标准就会变化：

- 更看重 Wayland 下的窗口行为
- 更看重 Linux 原生桌面适配
- 更看重长期运行稳定性
- 更看重高频图表、日志、串口通信时的可控性

在这个前提下，推荐顺序应调整为：

1. **Qt 6 + QML + QSerialPort**
2. **Rust + GTK4 / libadwaita**
3. **Rust + Tauri**

---

## 6. 为什么 Wayland 优先时更推荐 Qt 6 + QML + QSerialPort

这是当前最推荐的方案。

### 6.1 Qt 对 Wayland / X11 都有正式支持

Qt 在 Linux 上本身就有面向 **Wayland** 和 **X11** 的平台插件机制。

这意味着如果一开始用 Qt：

- 可以把 Wayland 当主平台做
- 后续补 X11 时不需要重做 UI 框架
- 更多是在平台适配和细节验证层面补齐

---

### 6.2 QSerialPort 很适合串口工具

串口工具是典型的：

- 本地 I/O 密集型
- GUI 与通信紧耦合
- 长时间运行
- 对稳定性要求高

Qt 在这种场景下非常成熟，`QSerialPort` 是官方模块，适合作为长期产品底层。

---

### 6.3 QML 适合做复杂交互界面

类 JCom 工具后面通常会有：

- 多面板布局
- 终端窗口
- 协议字段编辑面板
- 实时状态区
- 图表区
- 日志区
- 拖拽布局 / 分栏 / 标签页

QML/Qt Quick 在这些场景下，比传统 Widget 更适合承担主 UI。

特别是：

- 滚动
- 动画
- 缩放
- 自定义绘制
- 实时视图

这些能力都更自然。

---

## 7. 为什么 GTK4 / libadwaita 是第二选择

如果特别强调：

- Linux 原生桌面感
- Wayland 生态贴合
- GNOME 风格一致性
- 暂时不急着做 Windows/macOS

那 GTK4 / libadwaita 也是很好的选择。

### 优点

- Wayland 生态中原生感更强
- 与 GNOME 桌面集成自然
- 更贴近现代 Linux 桌面风格

### 不放第一的原因

因为这个项目并不是普通 Linux 小工具，而是希望做成：

> 类 JCom 的工程调试平台

这种工具通常会演化出：

- 协议编辑器
- 复杂图表
- 属性面板
- 多布局
- 工程文件管理
- 长期维护的 UI 体系

从长期复杂度和跨平台可能性来看，Qt 更稳。

---

## 8. 为什么 Tauri 在这个项目里排第三

Tauri 不是不能做，而是在当前约束下不再是首推。

### 主要原因

#### 8.1 Linux 上依赖 WebKitGTK 生态

Tauri 在 Linux 上依赖 `webkit2gtk` / WebView 生态。

这意味着：

- 对系统依赖更敏感
- 各发行版环境差异会更明显
- 工具部署与兼容性会更复杂

对于工程师工具来说，这不是最理想的情况。

---

#### 8.2 串口工具是高频本地工程应用

串口工具常见场景包括：

- 高频日志刷屏
- 实时曲线
- 多面板联动
- 大量配置编辑
- 长时间挂机运行

这些场景并不是 Web 技术做不了，而是从“轻、稳、可控”的角度看，Qt 往往更省心。

---

#### 8.3 Wayland 下窗口管理能力天然更受限

Wayland 的设计里，很多窗口行为是 compositor 决定的，不是应用可以完全掌控的。

因此：

- 不应依赖绝对窗口坐标
- 不应依赖大量悬浮窗
- 不应把复杂交互建立在窗口管理黑科技上

如果使用“桌面壳 + Web 前端”架构，定位和排查这类边缘问题通常会更绕。

---

## 9. Wayland 优先时必须提前接受的设计约束

这一部分比技术选型本身更重要。

---

### 9.1 不要依赖绝对窗口控制

Wayland 下不应把产品设计建立在这些前提上：

- 启动时强行摆放窗口到某个屏幕坐标
- 精准控制多窗口布局
- 严重依赖全局桌面坐标系
- 把关键功能放在很多悬浮小窗上

### 建议做法

- 单主窗口优先
- 窗口内部多 Dock / Split / Tab
- 工具面板以内嵌为主
- 保存“逻辑布局”，不要过度保存“物理像素位置”

这既适合 Wayland，也利于未来跨平台。

---

### 9.2 必须从第一天开始考虑高 DPI / 分数缩放

Wayland 下缩放不是小问题，而是架构问题。

需要保证以下内容在不同缩放倍率下都正常：

- 字体
- 终端显示
- 表格行高
- 图表渲染
- 图标
- 自绘控件命中区域
- 光标与拖拽交互

### 具体要求

工具在以下倍率下至少应验证：

- 100%
- 125%
- 150%
- 175%
- 200%

---

### 9.3 不要把系统托盘当核心入口

托盘可以做，但不应该承担核心功能。

关键操作必须都能在主窗口完成，比如：

- 打开串口
- 发送命令
- 查看连接状态
- 查看日志
- 切换工程

原因是不同 Linux 桌面环境在这类细节上的行为并不统一。

---

### 9.4 设备权限和热插拔要产品化

Linux 下串口工具很常见的实际问题：

- `/dev/ttyUSB0` 权限不足
- 用户不在 `dialout` 等组
- 热插拔后设备名变化
- 不同 USB-Serial 芯片枚举表现不同

这些问题不能只是写在 README 里，应该做成正式能力。

### 建议 UI 直接支持

- 权限检查
- 无权限时给出解决建议
- 记录 VID / PID / Serial Number，而不只记录设备路径
- 自动重连策略
- 断开 / 重连日志
- 热插拔状态提示

---

## 10. 推荐的整体技术架构（Qt 路线）

推荐方案：

> **Qt 6 + QML + QSerialPort**

### 10.1 Core 层

这一层纯逻辑，不依赖 UI：

- 串口会话
- 数据读写
- 协议编解码
- 校验算法
- 日志系统
- 会话回放
- 工程文件读写
- 设备识别和自动重连

---

### 10.2 Service 层

负责把核心能力暴露给 UI：

- `PortManager`
- `SessionManager`
- `ProtocolManager`
- `ChartService`
- `WorkspaceService`

---

### 10.3 UI 层（QML）

负责：

- 主窗口布局
- 终端面板
- 字段面板
- 命令模板面板
- 图表
- 日志窗口
- 工程保存 / 加载
- 状态提示和错误提示

---

## 11. 为什么推荐 QML，而不是 Widgets

虽然 Qt Widgets 也能做，但不建议它作为主战场。

### 原因

因为类 JCom 工具通常会有：

- 实时曲线
- 大量状态刷新
- 终端滚动
- 动态面板
- 可视化协议编辑
- 响应式布局

这些场景下，QML/Qt Quick 更适合。

### 建议方式

- 业务逻辑：C++（必要时可混合 Rust）
- UI：QML
- 不建议以 QWidget 作为主要界面框架

---

## 12. 推荐的开发阶段划分

---

### 阶段 1：Linux / Wayland MVP

先建立系统骨架，目标不是漂亮，而是稳定。

#### 功能范围

- 串口枚举
- 打开 / 关闭
- HEX / ASCII 收发
- 周期发送
- 发送列表
- 基础日志
- 自动重连
- 工程保存

#### Wayland 验收项

- KDE Plasma Wayland 下可用
- GNOME Wayland 下可用
- 125% / 150% 缩放正常
- USB 串口热插拔状态正常
- 多显示器切换后不崩
- 主窗口布局恢复正常

---

### 阶段 2：协议系统

目标是从“串口助手”升级成“协议工具”。

#### 功能

- 自定义帧结构
- 字段配置
- 自动 checksum / CRC
- 接收解析
- 表格化展示
- 简单实时曲线

---

### 阶段 3：控件化交互

这是类 JCom 的核心体验来源之一。

#### 功能

- 按钮
- 开关
- 滑条
- 枚举选择
- 命令模板绑定
- 一键初始化流程

做到这里，产品已经很接近目标形态。

---

### 阶段 4：补 X11

等 Wayland 版本稳定后，再补 X11。

这一步更多是：

- 测试窗口行为
- 修复字体、缩放、剪贴板等细节
- 处理少量平台差异

而不是重构整个产品。

---

## 13. 协议模型设计建议

真正的壁垒不在串口收发，而在协议抽象是否做得对。

建议至少抽象以下对象。

---

### 13.1 PortSession

表示一个串口会话：

- `portName`
- `baudRate`
- `dataBits`
- `stopBits`
- `parity`
- `flowControl`
- `autoReconnect`
- `lineEnding`
- `timestampMode`

---

### 13.2 PacketSchema

表示一个协议模板：

- `name`
- `framing`
- `fields[]`
- `checksumRule`
- `encodeRules`
- `decodeRules`

---

### 13.3 FieldDef

每个字段定义：

- `key`
- `label`
- `type`
- `length`
- `endian`
- `signed`
- `scale`
- `unit`
- `formula`
- `visible`
- `editable`
- `chartable`

---

### 13.4 CommandTemplate

一条可发送命令：

- `schemaRef`
- `defaultValues`
- `uiWidgetMappings`
- `sendMode`

---

### 13.5 ParserPipeline

接收解析流程：

- 字节流接收
- 帧边界识别
- 协议解码
- 派生字段计算
- 校验
- 事件分发
- 图表 / 日志 / 存储

---

### 13.6 Workspace / Project

工程文件：

- 串口配置
- 命令模板
- 协议 schema
- 图表布局
- 面板布局
- 日志设置

---

## 14. 工程文件和协议配置格式建议

### 推荐做法

- **YAML**：用于人可读的协议 schema
- **JSON**：用于运行时配置和工程持久化
- **SQLite**：用于大型工程、日志索引、会话记录
- **CSV**：用于日志导出
- **原始二进制**：用于抓包保存与回放

### 协议定义示意

```yaml
name: imu_frame
framing:
  type: header_length
  header: "AA55"
fields:
  - key: seq
    type: u16
    endian: little
  - key: ax
    type: i16
    scale: 0.001
    unit: g
    chart: true
  - key: ay
    type: i16
    scale: 0.001
    unit: g
    chart: true
  - key: az
    type: i16
    scale: 0.001
    unit: g
    chart: true
checksum:
  type: crc16_modbus
  range: all_before_checksum
